# NexusAZ — Database Layer

Production-sensible PostgreSQL + PostGIS data layer for **NexusAZ**, a
Just-in-Time vessel arrival intelligence platform centered on the Baku
commercial port.

## Folder layout

```
db/
├── __init__.py
├── alembic.ini
├── alembic/
│   ├── env.py
│   ├── script.py.mako
│   └── versions/
│       └── 20250101_0001_initial_schema.py
├── database.py          # async engine + session + Base
├── models.py            # SQLAlchemy 2.0 ORM models
├── schema.sql           # canonical raw SQL DDL
├── seed.sql             # realistic seed dataset (Baku port)
├── requirements.txt
└── README.md            # ← this file
```

## Stack

- PostgreSQL 15+ with **PostGIS 3.x** and **pgcrypto**
- SQLAlchemy **2.0 async** + **asyncpg**
- **GeoAlchemy2** for spatial ORM types
- **Alembic** for migrations

## Quick start

```powershell
# 1. Install deps
pip install -r db/requirements.txt

# 2. Set connection string (defaults to localhost)
$env:NEXUSAZ_DATABASE_URL = "postgresql+asyncpg://user:pass@localhost:5432/nexusaz"

# 3a. Apply via Alembic (recommended)
alembic -c db/alembic.ini upgrade head

# 3b. Or apply raw SQL directly
psql $env:NEXUSAZ_DATABASE_URL -f db/schema.sql

# 4. Load seed data
psql $env:NEXUSAZ_DATABASE_URL -f db/seed.sql
```

## Entities

| `users`               | Auth + RBAC (admin, operator, analyst, viewer)       |

| `esg_metrics`         | Fuel/CO₂/NOx/SOx + waiting-time per voyage           |

## Schema optimization notes (beyond the spec)

These are the engineering choices added on top of the bare requirements
and **why they matter for a maritime telemetry MVP**:

1. **`BIGSERIAL` PK on `vessel_positions`.** AIS feeds easily produce
   millions of rows/day. UUIDs would inflate the time-series table and
   wreck B-tree locality. A monotonically increasing 8-byte integer keeps
   the hot insertion path tight and indexes small.

2. **Composite index `(vessel_id, recorded_at DESC)`.** The dominant query
   on AIS data is *"give me the last N positions for vessel X"*. A
   composite descending index serves this directly without a sort step.

3. **Combined GIST index `(position, recorded_at)`.** Spatio-temporal
   queries (*"vessels within polygon P during last 24h"*) are the bread
   and butter of port intelligence. `btree_gist` lets PostGIS combine the
   spatial filter with the time predicate inside a single index scan.

4. **`(vessel_id, eta DESC)` and `(berth_id, eta)` on `port_bookings`.**
   Two access patterns dominate: *next berth for a vessel* and *berth
   utilization timeline*. Composite indexes turn both into index-only
   scans.

5. **Enum-as-CHECK instead of native ENUM.** Native PG enums are rigid
   (altering values requires `ALTER TYPE`). For a fast-evolving MVP, a
   `VARCHAR + CHECK` is friendlier to migrations while still rejecting
   invalid values at write time.

6. **`gen_random_uuid()` server-side defaults.** All UUID PKs are
   generated by the database via `pgcrypto`, so application code can
   `INSERT … RETURNING id` without round-tripping a Python-side UUID.

7. **Range constraints on `cog_deg` / `heading_deg` (0–360)** and on
   `confidence` (0–1) and `period_end >= period_start`. Bad AIS data is a
   real problem; rejecting it at the boundary protects every downstream
   analytic.

8. **`ON DELETE CASCADE` on telemetry/bookings/recommendations/ESG**, but
   `ON DELETE SET NULL` on `port_bookings.created_by`. Operational data
   should follow the vessel/booking lifecycle; user references should not
   delete bookings when an employee is offboarded.

9. **`updated_at` trigger via `set_updated_at()`.** Keeps audit columns
   honest even when writers forget to set them — important for
   regulatory ESG reporting.

10. **`pool_pre_ping=True` + `pool_recycle=1800`** in `database.py`.
    Long-lived async workers behind NAT/load-balancers regularly suffer
    silent TCP drops; pre-ping avoids the classic *"server closed the
    connection unexpectedly"* error in production.

11. **Explicit named GIST indexes.** GeoAlchemy2 auto-creates spatial
    indexes, but giving them stable names (`ix_berths_location`,
    `ix_vessel_positions_geom`) makes Alembic diffs deterministic and
    operational tooling (`REINDEX`, `pg_stat_statements`) much easier.

12. **`expire_on_commit=False`** on the session factory. Async handlers
    typically serialize ORM objects *after* commit; the default behavior
    would trigger lazy reloads on a closed session.

13. **PostGIS extension is *not* dropped on `downgrade()`.** Extensions
    are global to the database and may be shared with other schemas.
    Dropping them is opt-in and commented in the migration.
